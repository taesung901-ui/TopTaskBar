# TopTaskBar 런처 구현 계획

## 1. 목표

상단 바의 `T` 버튼을 눌렀을 때, 작업표시줄의 간단한 런처처럼 동작하는 앱 실행 패널을 아래로 펼친다.

원하는 동작:
- 현재 실행 중 앱 버튼 구역의 맨 왼쪽에 `T` 버튼을 둔다.
- `T` 버튼을 클릭하면 런처 팝업이 열린다.
- 런처 팝업에는 실행 가능한 앱 목록이 보인다.
- 앱을 클릭하면 해당 프로그램이 실행된다.
- 바깥을 클릭하면 팝업이 닫힌다.

이 기능은 현재의 “실행 중 창 전환” 기능과는 별개의 기능이다.

## 2. 현재 구현 방향

현재 버전은 다음 방향으로 간다.

- 자동 수집은 하지 않는다.
- 앱 목록은 처음부터 JSON 기반으로 관리한다.
- 기본 앱 목록은 설정 파일 기본값으로 생성한다.
- 런처 UI와 팝업 동작을 먼저 안정화한다.

즉, 하드코딩 목록을 잠깐 쓰고 나중에 JSON으로 옮기는 방식이 아니라, 처음부터 JSON 구조를 기준으로 설계한다.

## 3. 자동 수집을 바로 하지 않는 이유

설치된 프로그램 자동 수집은 생각보다 복잡하다.

문제점:
- 시작 메뉴, 레지스트리, 바로가기, 실행 파일 경로가 제각각이다.
- 중복 항목이 생기기 쉽다.
- 실행 가능한 경로를 얻기 어려운 경우가 있다.
- UWP / 스토어 앱은 실행 방식이 다르다.
- 사용자에게 어떤 항목을 보여줄지 정제 규칙이 필요하다.

지금 먼저 검증해야 할 것은 다음이다.

- `T` 버튼 위치가 적절한가
- 팝업 열기/닫기 UX가 좋은가
- 앱 목록 표시 방식이 적절한가
- 클릭 시 실행 흐름이 자연스러운가
- 아이콘이 잘 보이는가

따라서 현재 단계에서는 자동 수집보다 “좋은 런처 인터페이스를 만드는 것”이 더 중요하다.

## 4. 1차 구현 범위

### 포함

- 앱 버튼 구역 맨 왼쪽에 `T` 버튼 배치
- `T` 버튼 아래에 런처 팝업 표시
- 앱 목록을 JSON 설정에서 읽기
- 설정 파일이 없거나 앱 목록이 비어 있으면 기본 앱 목록 생성
- 앱 아이콘과 이름 표시
- 클릭 시 프로그램 실행
- 실행 후 팝업 닫기
- 바깥 클릭 시 팝업 닫기
- 달력 팝업과 동시에 하나만 열리도록 조정
- 파일 선택으로 앱 추가

### 제외

- 설치된 프로그램 자동 검색
- 검색 기능
- 앱 삭제 UI
- 드래그 정렬
- 카테고리
- 최근 실행 앱
- UWP / 스토어 앱 지원
- 바로가기(`.lnk`) 지원
- 폴더 / URL 실행 지원

## 5. UI 방향

런처는 현재 상단 바의 일부처럼 보여야 한다.

권장 레이아웃:
- `T` 버튼은 현재 실행 중 앱 버튼 구역의 맨 왼쪽
- 팝업은 `T` 버튼 바로 아래에 정렬
- 둥근 모서리 패널
- 리스트 중심 레이아웃
- 왼쪽 아이콘, 오른쪽 앱 이름
- 기존 TopTaskBar와 같은 테마 흐름 유지

권장 동작:
- `T` 버튼 클릭 시 런처 팝업 토글
- 런처를 열면 달력 팝업 닫기
- 달력을 열면 런처 팝업 닫기
- 바깥 클릭 시 런처 팝업 닫기
- 앱 클릭 시 실행 후 팝업 닫기

## 6. 데이터 구조

앱 하나를 표현하는 런처 항목 모델이 필요하다.

### 런처 표시용 모델

필드:
- `Name`
- `Path`
- `Arguments`
- `WorkingDirectory`
- `Icon`

파일:
- `TopTaskBar/LauncherAppItem.cs`

### 설정 저장용 모델

필드:
- `Name`
- `Path`
- `Arguments`
- `WorkingDirectory`

파일:
- `TopTaskBar/LauncherAppSetting.cs`

## 7. 설정 구조

기존 설정 파일에 런처 앱 목록을 같이 저장한다.

`TopTaskBarSettings`에 추가할 항목:
- `List<LauncherAppSetting> PinnedApps`

수정 파일:
- `TopTaskBar/TopTaskBarSettings.cs`
- `TopTaskBar/SettingsStore.cs`

## 8. 초기 JSON 방향

앱 목록은 처음부터 JSON에 들어가도록 한다.

예시 구조:

```json
{
  "windowSlotWidth": 42,
  "pinnedApps": [
    {
      "name": "Explorer",
      "path": "C:\\Windows\\explorer.exe",
      "arguments": "",
      "workingDirectory": "C:\\Windows"
    },
    {
      "name": "Notepad",
      "path": "C:\\Windows\\System32\\notepad.exe",
      "arguments": "",
      "workingDirectory": "C:\\Windows\\System32"
    }
  ]
}
```

중요한 점:
- 처음부터 이 구조로 저장한다.
- 기본 설정 파일 생성 시 `PinnedApps`도 함께 채운다.
- 이후 UI에서 앱 추가/삭제 기능이 생겨도 같은 구조를 유지한다.

## 9. 앱 실행 방식

앱 실행은 `Process.Start`와 `UseShellExecute = true`를 사용한다.

이유:
- 단순하다
- 일반적인 Windows 데스크톱 앱 실행에 잘 맞는다
- 나중에 `.lnk`, 폴더, URL까지 확장하기 쉽다

1차 지원 대상:
- `.exe` 경로

## 10. 아이콘 처리 전략

1차에서는 앱 경로에서 아이콘을 읽어온다.

방향:
- exe 파일 아이콘 추출
- `ImageSource`로 변환
- 경로가 잘못됐으면 기본 아이콘 또는 fallback 문자 표시

파일:
- `TopTaskBar/AppIconHelper.cs`

이 로직은 `WindowCatalog`와 분리한다.

## 11. 팝업 충돌 방지 규칙

런처와 달력은 동시에 열리지 않게 한다.

규칙:
- 런처를 열면 달력 닫기
- 달력을 열면 런처 닫기
- 동시에 하나의 팝업만 유지

## 12. 구현 단계

현재 구현은 다음처럼 나누는 것이 가장 안정적이다.

### Phase L1: 런처 팝업 골격

목표:
- `T` 버튼을 앱 버튼 구역 왼쪽으로 이동
- `T` 버튼 아래에 런처 팝업 추가
- 팝업 열기/닫기 동작 완성
- 달력 팝업과 충돌 없게 정리

### Phase L2: JSON 기반 앱 목록

목표:
- `PinnedApps` 설정 구조 추가
- 설정 파일에서 앱 목록 읽기
- 앱 목록이 없으면 기본값 생성
- 런처 목록과 UI 바인딩 연결

### Phase L3: 실행 기능 연결

목표:
- 앱 클릭 시 `Process.Start` 실행
- 실행 후 팝업 닫기
- 실패 시 예외 처리

### Phase L4: 아이콘 적용

목표:
- exe 경로에서 아이콘 추출
- 런처 목록에 아이콘 표시

### Phase L5: 파일 선택으로 앱 추가

목표:
- 런처 팝업 하단에 `앱 추가` 버튼 추가
- 파일 선택 대화상자로 `.exe` 선택
- 이름/작업 폴더 자동 채움
- JSON에 저장 후 즉시 목록 갱신
- 중복 추가 방지

## 14. 실행 중 앱에서 런처에 추가 기능

이 기능은 고려할 가치가 높다.  
다만 “현재 창 상태 저장”처럼 보이면 안 되고, 정확한 의미는 아래와 같다.

정의:
- 현재 실행 중인 창에서
- 그 창이 속한 **프로그램의 실행 정보**를 추출해
- 런처에 추가하는 기능

즉 저장되는 것은:
- 현재 창 상태가 아니라
- 실행 파일 정보이다.

## 15. 이 기능의 위험 요소

추가는 되었는데 사용자가 기대한 방식과 다를 수 있다.

예:
- 현재 창에서 열어둔 문서/프로젝트 상태는 저장되지 않음
- 같은 exe가 여러 역할을 가지는 앱은 기대와 다르게 열릴 수 있음
- 일부 시스템 앱 / 특수 앱은 실행 경로를 얻기 어렵거나 재실행이 불안정할 수 있음
- UWP / 스토어 앱은 1차 지원 대상이 아님

따라서 이 기능은 “현재 창 저장”이 아니라
**“현재 실행 중인 프로그램을 빠르게 런처에 추가”** 로 정의해야 한다.

## 16. 추천 안전장치

이 기능을 넣을 때는 아래 안전장치를 반드시 같이 둔다.

### 안전장치 A: 메뉴 문구

권장 문구:
- `런처에 앱 추가`
- `이 프로그램을 런처에 추가`

비추천 문구:
- `현재 창 저장`
- `이 창 고정`
- `작업 저장`

### 안전장치 B: 중복 방지

기준:
- `Path`가 같으면 중복으로 간주

동작:
- 이미 있으면 추가하지 않음
- 메시지:
  - `이미 런처에 추가된 앱입니다.`

### 안전장치 C: 경로 획득 실패 처리

동작:
- 추가 중단
- 메시지:
  - `이 앱의 실행 경로를 가져오지 못했습니다.`

### 안전장치 D: 기대치 관리

의미를 명확히 해야 한다.

권장 설명:
- 현재 창 상태가 아니라 프로그램 실행 정보가 추가된다

### 안전장치 E: 지원 범위 제한

1차 지원 범위:
- 일반 데스크톱 앱
- 안정적으로 exe 경로를 얻을 수 있는 앱

지원 제외 또는 실패 처리 가능 대상:
- UWP / 스토어 앱
- 일부 관리자 권한 앱
- 일부 시스템 특수 창

## 22. Next 3 상세 계획

`Next 3`는 런처가 단순한 exe 실행기를 넘어서,
자주 여는 폴더 / 웹페이지 / 바로가기까지 담을 수 있게 확장하는 단계다.

이 단계의 핵심 원칙은 아래와 같다.

- 폴더와 URL은 `JSON 수정` 버튼을 통해 직접 넣을 수 있게 한다.
- `.lnk`는 사용자가 경로를 손으로 쓰지 않도록 `앱 추가` 버튼에서 선택 가능하게 한다.
- 내부 저장 구조는 그대로 `PinnedApps`를 사용한다.
- 타입 판별은 `Path` 값으로 한다.

구분 기준:
- `.exe` 경로면 실행 파일
- 디렉터리 경로면 폴더
- `http://` 또는 `https://`면 URL
- `.lnk` 경로면 Windows 바로가기

## 23. 사용자 관점의 사용 방식

### 폴더
- 사용자는 `settings.json`에 자주 여는 폴더 경로를 넣는다.
- 런처에서 누르면 탐색기로 해당 폴더가 열린다.

### URL
- 사용자는 `settings.json`에 웹 주소를 넣는다.
- 런처에서 누르면 기본 브라우저로 해당 주소가 열린다.

### `.lnk`
- 사용자는 `앱 추가` 버튼을 눌러 파일 선택 창에서 바탕화면 또는 시작 메뉴의 바로가기를 고른다.
- 런처는 그 `.lnk`를 저장하고, 나중에 누르면 Windows 바로가기 실행 방식으로 동작하게 한다.

즉 `.lnk`는 JSON 직접 입력보다
`앱 추가 -> 바로가기 선택` 흐름을 기본 UX로 삼는다.

- 경로/문자열이 어떤 타입인지 판별하는 공통 함수 추가
- 타입별 유효성 검사 분리

### Phase N3-2: 폴더 실행 지원

목표:
- `PinnedApps.Path`가 디렉터리면 탐색기로 열기

핵심 작업:
- 런처 클릭 실행 로직에서 폴더를 별도 분기 없이 안정적으로 열리게 정리
- 아이콘은 폴더용 기본 아이콘 또는 fallback 사용

사용자 입력 방식:
- `JSON 수정`에서 폴더 경로 직접 입력

### Phase N3-3: URL 실행 지원

목표:
- `PinnedApps.Path`가 URL이면 기본 브라우저로 열기

핵심 작업:
- URL 판별 함수 추가
- `Process.Start` + `UseShellExecute=true`로 실행
- URL 전용 아이콘/fallback 처리

사용자 입력 방식:
- `JSON 수정`에서 URL 직접 입력

### Phase N3-4: `.lnk` 선택 지원

목표:
- `앱 추가` 버튼에서 `.exe`뿐 아니라 `.lnk`도 고를 수 있게 변경

핵심 작업:
- 파일 선택 대화상자 필터 확장
  - `*.exe;*.lnk`
- 선택한 파일이 `.lnk`면 그대로 `PinnedApps`에 저장
- 이름 기본값은 파일명 기준으로 채움

사용자 입력 방식:
- `앱 추가` 버튼 클릭
- 바탕화면/시작 메뉴의 `.lnk` 선택

### Phase N3-5: `.lnk` 실행 지원

목표:
- 런처에서 `.lnk` 항목을 누르면 Windows 바로가기처럼 실행

핵심 작업:
- `.lnk` 파일 경로를 `Process.Start`로 실행
- 필요 시 아이콘 추출 fallback 검토

### Phase N3-6: 예외 처리와 메시지 정리

목표:
- 타입별 실패 메시지 정리

예:
- 존재하지 않는 폴더
- 잘못된 URL
- 끊어진 `.lnk`
- 접근 불가 경로

## 28. 검색 기능 설계

검색 기능은 런처 팝업 안에 입력창 하나를 두고,
현재 런처 목록을 즉시 필터링하는 방식으로 구현한다.

정의:
- 새로운 앱을 찾는 검색이 아니라
- 현재 `PinnedApps` / `LauncherApps` 목록을 거르는 필터 기능

사용자 관점:
- 런처를 열고 검색창에 일부 이름을 입력한다
- 입력한 문자열과 맞는 항목만 아래 목록에 남는다
- 항목을 클릭하면 기존처럼 실행된다

예:
- `ch` 입력 -> `Chrome`
- `doc` 입력 -> `! Documents`
- `goo` 입력 -> `! Google`

UI 방향:
- 런처 상단에 `Launcher` 제목
- 그 아래 검색 입력창
- 그 아래 최근 실행 아이콘 영역
- 그 아래 전체 목록

검색 규칙:
- 이름 기준 부분 일치
- 대소문자 구분 없이 비교
- 필요하면 나중에 경로까지 확장 가능하지만 1차는 이름만 검색

## 30. 최근 실행 앱 설계

최근 실행 앱은 시스템 전체 기준이 아니라
**런처를 통해 실행한 항목만** 기록하는 방식으로 단순하게 구현한다.

정의:
- 사용자가 런처에서 어떤 항목을 눌러 실행했을 때
- 그 항목을 최근 실행 목록의 맨 앞으로 올린다

기록 규칙:
- 중복이면 기존 항목 제거 후 맨 앞으로 이동
- 최대 3~5개만 유지
- 1차는 5개를 기본값으로 고려

사용자 관점:
- 런처를 열면 상단에 최근 실행한 항목 아이콘만 보인다
- 아이콘만 3~5개 정도 간단히 배치
- hover 시 이름 툴팁 표시
- 클릭 시 해당 항목을 다시 실행

UI 방향:
- 검색 입력창 아래
- 전체 목록 위
- 최근 실행 아이콘 영역은 작고 가볍게 유지

타이머/알람은 추후 기능 추가가 예정되어 있으므로

별도 클래스로 분리하고, `MainWindow`는 화면 연결만 담당한다.

## 43. 알람 기능 방향

알람은 기존 타이머 도구 팝업을 확장하는 형태로 구현한다.

핵심 방향:
- 타이머/알람 팝업 내부를 탭 구조로 변경
- 상단 탭:
  - `타이머`
  - `알람`
- 타이머는 기존 구현을 유지
- 알람은 새 탭에서 별도 UI와 상태를 가짐

1차 알람 범위:
- 단일 알람 1개
- 시/분 설정
- 알람 설정
- 알람 해제
- 현재 상태 표시
- 시간이 되면 완료 알림 표시

1차에서 제외:
- 여러 개 알람
- 반복 알람
- 요일 반복
- 스누즈
- 소리 선택

## 44. 사용자 관점의 사용 방식

사용자 관점:
- 런처 하단 `타이머` 버튼을 누른다.
- 도구 팝업 상단 탭에서 `알람`을 선택한다.
- 시/분을 입력하거나 올림/내림 버튼으로 조정한다.
- `설정`을 누르면 다음 도래 시각 기준으로 알람이 예약된다.
- `해제`를 누르면 현재 알람이 취소된다.
- 시간이 되면 중앙 알림 창이 뜬다.

예:
- 현재 시간이 오전 9:10일 때 `09:30`으로 설정
  - 오늘 09:30 알람 대기
- 현재 시간이 오전 9:40일 때 `09:30`으로 설정
  - 내일 09:30 알람 대기

## 52. 알람 기능 업그레이드 방향

다음 단계의 알람은 단일 알람 1개 구조에서 벗어나,
안드로이드 시계 앱처럼 "알람 목록 + 토글 + 편집 팝업" 구조로 확장한다.

핵심 방향:
- 알람 탭 메인 화면은 편집 화면이 아니라 알람 카드 목록이 된다.
- 기본 알람 3개를 미리 생성해 둔다.
- 각 알람 카드 우측에 토글을 두어 빠르게 활성화/비활성화한다.
- 카드 본문을 클릭하면 별도 편집 팝업이 열린다.
- 편집 팝업에서는 오전/오후, 시, 분, 요일을 수정한다.
- 요일을 하나도 선택하지 않으면 1회성 알람으로 간주한다.

## 53. 사용자 관점 동작 방식

사용자 흐름:
- 알람 탭을 열면 기본 알람 3개가 카드 형태로 보인다.
- 자주 쓰는 알람은 토글만 켜서 바로 활성화한다.
- 시간이나 요일을 바꾸고 싶으면 카드 본문을 클릭한다.
- 편집 팝업에서 오전/오후, 시, 분, 요일을 바꾸고 저장한다.
- 요일을 선택하지 않은 알람은 다음 도래 시각에 한 번만 울린다.
- 요일을 선택한 알람은 선택한 요일에 반복된다.

목표 UX:
- 빠른 on/off는 목록에서 해결
- 상세 수정은 별도 팝업에서 해결
- 목록 화면은 간단하고 시인성 있게 유지

## 55. 요일 정책

- 반복 알람:
